API

What is an API?

API stands for Application Program Interface. What they provide can vary greatly from host to host. Generally, APIs provide access to information or functionality that the host has developed or owns, while abstracting the underlying implementation. Each API has clearly defined protocols and tools to shape how you interact with and access their data. Developers can build an application around one or more APIs making the development process much easier and more efficient.

Web APIs

Developers usually interact with web APIs via sending HTTP requests, then receiving a response of data in XML or JSON format. The developer then uses this data in their own applications, often resulting in the data reaching new and different users than would have otherwise been reached.

The Petfinder API gives access to their expansive database, as well as many options for gathering and using the information. We will get into the specifics of this in a little bit.

RESTful APIs

The Petfinder API is a RESTful API. The structure of a RESTful API is based on the REST architecture style. In order for an API to be RESTful it must abide by these six required constraints:

  • The client interface and server interface and storage are completely separate allowing both to be altered and scaled independently of each other. This means both you and the host are free to change different aspects on your respective ends without worrying about “breaking” the other end.

  • No client context is stored on the server between requests during client-server communication. The state of the session, as well as all the necessary information, is in the client request. If the session must be in a persistent state, such as for authentication, the session state can be transferred to a different, intermediary server.

    To transition to a new state, the client sends requests to the server. If there are one or more outstanding requests, the client is considered “in transition.” While a process may be comprised of many requests and states, each individual request can stand alone and is a potential end or beginning state to a process.

  • Both clients and intermediary servers can cache responses. Clients must define their responses as either cacheable or not. If a response has time sensitive data and is cached, the information may become stale. Being able to flag whether the response is cacheable or not can eliminate excess client-server interactions improving overall performance.

  • When a client is communicating with the API, they have no way of knowing if they are interacting with the main server or an intermediary server. This enables load balancing, which is more efficient, and helps in terms of security.

  • This is the only constraint that could be considered optional. While it may never be utilized, a server can transfer executable code to the client, temporarily changing the client’s functionality.

  • The REST system design is reliant on the idea of individual functionality of each part of the whole. This can only be achieved if every aspect of the interface is uniform. Because each part is capable of being autonomous, they can change without affecting the other parts. We could go into great detail as to how this is implemented, but this is a little outside the scope of this guide.

The most common way to use RESTful APIs is to use HTTP protocol using GET, POST, PUT, and DELETE methods. The Petfinder API currently only uses the GET method, so that is where our focus will lie.
